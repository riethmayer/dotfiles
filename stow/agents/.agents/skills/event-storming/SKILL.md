# Event Storming

Run Brandolini's EventStorming as a chat conversation with the user. Default to the Big-Picture level — exploring the business end-to-end with just events, hotspots, systems, and people — and escalate to Process Modeling or Software Design only when the user asks for it.

The output is a **whiteboard**, not a static diagram. The skill emits a `.tldr` file — tldraw's native format — that opens cleanly in real tldraw with full pan/zoom/drag/edit. The user can rearrange stickies, draw their own arrows, branch new paths off the canvas. Static SVG was tried; it lost too much of what makes a storm useful.

## The three levels (and which one to start at)

Brandolini's progression — same artifact, same colors, just zooming in:

| Level | Adds | What you're looking for |
|---|---|---|
| **1. Big Picture** *(default)* | events + hotspots + systems + people | conflicts, goals, blockers, bounded-context seams |
| **2. Process Modeling** | + commands, policies, read models | value proposition, explicit policies, personas, individual goals |
| **3. Software Design** | + aggregates | aggregate boundaries, policies tied to aggregates, IDs |

**Always start at Big Picture.** It's the cheapest, broadest pass and almost always what the user actually wants when they say "let's storm this". The reason: discovery is supposed to surface disagreement, and the colors you add at Process / SD level start to demand consensus the user hasn't earned yet.

Escalate when the user explicitly signals:
- *"now add commands and policies"* / *"let's process-model this"* → Level 2
- *"now let's design the code"* / *"where are the aggregates?"* → Level 3

If they don't signal, don't escalate. A clean Big-Picture storm is a finished artifact, not a stepping stone.

## What you produce, always

Two files in the user's working directory:

1. **`storm.json`** — the structured content. Schema in `references/schema.md`. The file the user tweaks: rename events, add hotspots, regroup aggregates, retag outcomes.
2. **`storm.tldr`** — the tldraw file, generated by:
   ```bash
   python scripts/build_tldraw.py storm.json --out storm.tldr
   ```

JSON is the source of truth; `.tldr` is a view. Don't hand-edit the `.tldr` file — regenerate from JSON.

### Opening the .tldr file

Three options, in order of preference:

1. **Local dev app** (best for iteration). From the skill directory: `cd tldraw-dev && npm install && npm run dev`. Drop the `.tldr` into `tldraw-dev/public/storms/` (or use the bundled examples) and visit http://localhost:5173. Sidebar switches between storms. See `tldraw-dev/README.md` for details.
2. **tldraw.com**. File ▸ Open file ▸ pick the `.tldr`. Free, no setup, but you lose the conversation context.
3. **Self-hosted tldraw**. If the user already runs their own, point them at the `.tldr`.

## The structure: happy path + branches

Each storm is a **happy path** (events that occur when everything goes right) plus **branches** that fork below it (alternative paths, failure modes, edge cases). Brandolini's "Rush to the Goal" strategy: build the baseline first, mark every alternative with a hotspot, then come back and explore them one at a time.

```jsonc
{
  "happy_path": [
    {"kind": "actor", "label": "Visitor"},
    {"kind": "event", "label": "Customer Registered"},
    ...
    {"kind": "event", "label": "Customer Happy", "outcome": "happy"}
  ],
  "branches": [
    {
      "from": 1,                                // happy-path index to fork from
      "label": "What if signup verification fails?",
      "beats": [
        {"kind": "hotspot", "label": "Email verification required?"},
        {"kind": "event",   "label": "Verification Failed", "outcome": "sad"}
      ]
    }
  ]
}
```

Tagging an event with `"outcome": "happy"` colors it green and adds a big 🤣; `"outcome": "sad"` colors it red with a big 😭. End every branch in an outcome — that's how the user (and you) can tell the storm is "done" per Brandolini's first rule: every path ends in a stable state.

## How the conversation goes

Detailed facilitation per level lives in:

- **`references/level-1-big-picture.md`** *(read this first, every time)* — the default workflow: chaotic exploration → enforce timeline → pin people + systems → capture hotspots.
- **`references/level-2-process-modeling.md`** — read when the user escalates: how to add commands, policies, read models; the closed-loop "grammar in action"; the policies spectrum.
- **`references/level-3-software-design.md`** — read when the user wants to design code: aggregate hunting, symmetries on the timeline, "ambiguity does not compile".

Cross-cutting references that apply at any level:

- **`references/grammar.md`** — color palette, past-tense rule, what's *not* a domain event.
- **`references/schema.md`** — `storm.json` field reference.

## Brandolini's game rules

A storm is "done" when:

1. Every path on the spine ends in a **stable state** — tag it with `outcome: "happy"` or `outcome: "sad"`.
2. The **color grammar is respected** end-to-end.
3. Every **stakeholder is reasonably happy** with their view.
4. Every **hotspot** has been addressed — answered inline or noted for follow-up.
5. *(Software-Design only)* Every **aggregate is coherent**.

The hotspot rule in particular is non-negotiable — a storm without hotspots is a workshop where people were too polite to disagree.

## The bundled examples

Four canonical storms, all in `assets/`:

| File | Level | What's in it |
|---|---|---|
| `ecommerce-bp` | Big Picture | Shop end-to-end (10 happy + 5 failure branches), no commands/policies/aggregates. |
| `ecommerce-process` | Process Modeling | Same shop with commands, policies, read models interleaved on the happy path. |
| `ecommerce-sd` | Software Design | Process spine + Cart / Checkout / Payment / Order / Shipment / Return aggregate bands. |
| `insurance-claims` | Process Modeling | P&C insurance: FNOL → triage → investigate → decide → pay, with fraud / denial / dispute / subrogation branches. Cross-domain proof. |

Each ships as `<name>.storm.json` + `<name>.tldr`. To render after editing the JSON:

```bash
python scripts/build_tldraw.py assets/ecommerce-bp.storm.json --out assets/ecommerce-bp.tldr
```

Copy whichever matches the user's scope, rename the title, edit the beats, regenerate the `.tldr`, reload in tldraw.

## Visual grammar in the rendered output

Each `.tldr` includes a **legend column** on the left so the reader can decode the colors at a glance:

| Sticky | Element | Voice / form |
|---|---|---|
| 🟠 Orange | Domain event | past tense |
| 🔵 Blue | Command *(Level 2+)* | imperative verb |
| 🟡 Yellow + 🧍 | Actor | role, not name |
| 🟣 Violet | Policy *(Level 2+)* | "whenever X then Y" |
| 🟢 Green | Read model *(Level 2+)* | screen / data the actor consults |
| ⬜ Grey | External system | third-party / black box |
| 🟪 Violet diamond | Hotspot | question, friction, conflict |
| 🟢 + 🤣 | Happy outcome | end of a happy path |
| 🔴 + 😭 | Sad outcome | end of a failure branch |
| Yellow band *(SD only)* | Aggregate | noun with a lifecycle, wraps a contiguous run of events |

Full reference and the why-the-grammar-matters reasoning: `references/grammar.md`.
